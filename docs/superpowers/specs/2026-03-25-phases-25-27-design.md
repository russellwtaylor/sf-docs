# Phases 25-27 Design Spec

## Overview

Three features that enhance filtering and discoverability in sf-docs:

- **Phase 25**: `--name-filter` CLI flag for glob-based filename filtering
- **Phase 26**: Client-side fuzzy search in HTML output using fuse.js
- **Phase 27**: `@tag` annotations in ApexDoc with CLI and UI filtering

All three filters (`--type`, `--name-filter`, `--tag`) AND together to progressively narrow the documented set.

## Architecture: Layered Filters

Each filter applies at a specific pipeline stage:

```
--type        → gates which scanners run (already implemented)
--name-filter → filters SourceFile results post-scan, pre-parse
--tag         → filters parsed metadata post-parse, pre-AI
```

Search (Phase 26) is orthogonal — a build-time index generation step plus static JS assets in the HTML output.

---

## Phase 25: `--name-filter` / `--glob` Flag

### Goal

Narrow documentation to files matching a name pattern (e.g., `--name-filter 'Order*'`).

### CLI

- New arg: `--name-filter <PATTERN>` on `GenerateArgs`
- Accepts shell glob syntax: `Order*`, `*Service`, `*Order*`
- Stored as `Option<String>`
- Uses the `globset` crate (from BurntSushi/ripgrep — battle-tested, supports brace expansion)
- No short flag (consistent with `--type`)

### Filtering Logic

- Applied immediately after each scanner returns its `Vec<SourceFile>`
- Matches against `SourceFile.filename` which scanners already set to the logical name (e.g., `OrderService`, `My_Flow`), avoiding double-extension issues with `Path::file_stem()`
- Non-matching files dropped before parsing — no AI credits wasted
- Helper method: `GenerateArgs::name_matches(&self, filename_stem: &str) -> bool`
  - Returns `true` if no filter specified or if stem matches the pattern

### Integration

- Combines with `--type`: type gates scanners, name-filter further narrows results
- Verbose mode prints the active filter pattern
- Works across all 9 metadata types consistently

## Phase 26: Client-Side Search in HTML Site

### Goal

Make the generated HTML site searchable by class, method, field, and flow name without a backend.

### Search Library

- **fuse.js** for fuzzy matching (~25KB minified, ~6KB gzipped)
- Embedded in the Rust binary as a `const &str`, written to `search.js` during generation
- No CDN dependency — fully offline-capable

### Build-Time Index

- Generated after all HTML pages are rendered
- Written to `search-index.json` in the site root
- Each entry: `{ title, type, folder, summary, url, tags }`
- Covers all 9 metadata types

### Static Assets

`search.js` in site root contains:
- Minified fuse.js library
- Search initialization: loads `search-index.json`, creates Fuse instance
- Search UI logic: renders results as a filtered list with links
- Fuse config: search on `title`, `summary`, and `tags` fields, threshold ~0.3
- Debounce search input by ~200ms to avoid sluggishness on large sites

### HTML Integration

- Search input (`<input>`) at the top of the sidebar, above navigation links
- `<script src="search.js">` tag at bottom of each page (relative path with up-prefix)
- `search.js` resolves `search-index.json` path relative to its own `<script>` element's `src` attribute (derive site root from the script path)
- Search results replace sidebar navigation while typing
- Clearing the input restores full nav
- Results show item title, metadata type badge, and link

### Scope

- HTML format only. No impact on Markdown output.

## Phase 27: Tag / Label Filtering

### Goal

Allow teams to categorize docs with `@tag` annotations and filter by tag in CLI and UI.

### Parsing

- Recognize `@tag <label>` in ApexDoc block comments (`/** ... */`)
- Applies to Apex classes and Triggers only (they share the ApexDoc comment format)
- Multiple `@tag` annotations per class/trigger supported (one per line)
- Stored as `Vec<String>` on `ClassMetadata` and `TriggerMetadata`
- Regex: `@tag\s+(\w[\w-]*)` — alphanumeric + hyphens only (e.g., `billing`, `order-management`). Dots, slashes, and spaces are intentionally rejected to keep tag values simple and safe for use in HTML `data-tags` attributes and JS click handlers

### CLI

- New arg: `--tag <LABELS>` with `value_delimiter = ','`
- Stored as `Vec<String>` on `GenerateArgs`
- Helper: `GenerateArgs::tag_matches(&self, tags: &[String]) -> bool`
  - Returns `true` if `--tag` not specified, or if any item tag matches any requested tag (OR logic)

### Filtering Logic

- Applied post-parse, pre-AI — parse needed to find `@tag` annotations, but non-matching items skip AI
- Non-Apex/Trigger types: excluded from output when `--tag` is specified (user filtering by tag expects only tagged results, not everything that lacks tag support)
- Tag matching is case-insensitive (`--tag Billing` matches `@tag billing`)

### HTML Output

- Page header: tag badges as small pills next to the title (styled like existing access modifier badges)
- Index page: tag pills on each entry
- Sidebar items get a `data-tags` attribute
- Clicking a tag pill in content area filters sidebar to matching items (JS in `search.js`)

### Markdown Output

- Tags rendered as a line below the title: `**Tags:** \`billing\`, \`integration\``

### Implementation Note

Tags are parsed from source comments and stored on metadata structs (`ClassMetadata`, `TriggerMetadata`). They are NOT added to AI documentation response structs (`ClassDocumentation`, `TriggerDocumentation`) — tags come from source parsing, not AI output.

### Filter Combination

Full pipeline: `--type` -> `--name-filter` -> parse -> `--tag` -> AI

All three filters AND together. Each progressively narrows the set of files that reach the AI documentation step.
